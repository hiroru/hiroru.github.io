Systemd: One service to rule them all
Oct 19, 2017
12 minutes read

Where all Baggins

Systemd is the new kid on the block of init systems for Linux that aimed to be the new standard and indeed it did.

It claims to be an extremely fast and integrated init system with aggressive parallelization capabilities, socket management, mount points manager, all using D-BUS and cgroups technologies. Most modern distros and OSes like Arch Linux, CoreOS, Debian, Fedora, Gentoo, Red Hat, Ubuntu, and so on use it a default init system, so its ambitions are confirmed.

In this post we will talk about it and the basics to get used with common systemd tools. So, don’t afraid and remember it’s a dangerous business, Sysops, going out your binary door. You step onto the code, and if you don’t keep your feet, there’s no knowing where you might be swept off to1.

Inside systemd’s deep

Let’s get a deeper view of what’s systemd and how it works. Systemd is the first binary start before the kernel, in fact, it’s the kernel itself who run it.

Systemd is an init system, and as an init system its purpose is to manage all user land components and services. To perform such a glorious task a huge bundle of binaries work together to manage almost everything from our systems.

They replaces the startup scripts from SysV, BSD init or OpenRC for unit Files and runlevels for Targets as long as manage common services such as login with logind, logs with journald, network with networkd, device manager with udev, scheduled execution (cron), locales, time and date and hostnames.

Take a look to the below image from Wikipedia2 where you can see almost every binary involved in systemd. Keep in mind that systemd is growing everyday, so may not be complete but enough for the scope of this post.

In this introduction post we will focus in the below component list:

  • systemd: the service itself.
  • systemctl: the user tool to manage systemd.
  • systemd-analyze: the systemd boot process analyser.
  • journald: the logger daemon.
Targets

Targets from systemd are slightly different from runlevels. Targets are named instead of numbered and , unlike runlevels, there’s the possibility to have multiple ones active at the same time.

A target file looks like this:

[Unit]
Description=NFS client services
Before=remote-fs-pre.target
Wants=remote-fs-pre.target

Wants=nfs-blkmap.service
After=nfs-blkmap.service

# GSS services dependencies and ordering
Wants=auth-rpcgss-module.service
After=rpc-gssd.service rpc-svcgssd.service gssproxy.service

[Install]
WantedBy=multi-user.target
WantedBy=remote-fs.target

systemctl list-units --type=target

Unit files

As we said after, systemd uses unit files that replace the traditional shell scripts used in other init systems. These unit files uses declarative programming language4 to describe its usage.

Every kind of system utility use a specific unit file type: service, socket, device, mount, target, path, timer (cron job), snapshot, slice and scope.

These unit files are commonly located in three different paths in our systems (may change in some distros):

  • /usr/lib/systemd/system/ for units provided by installed packages.
  • /etc/systemd/system/ for system-wide units placed/customized by the system administrator.
  • ~/.config/systemd/ for units defined or customized by system users.

Let’s analyse the content of a unit file by running systemctl cat SERVICE_NAME. The below file it’s from Debian’s sshd unit:

systemctl cat sshd.service

# /lib/systemd/system/ssh.service
[Unit]
Description=OpenBSD Secure Shell server
After=network.target auditd.service
ConditionPathExists=!/etc/ssh/sshd_not_to_be_run

[Service]
EnvironmentFile=-/etc/default/ssh
ExecStart=/usr/sbin/sshd -D $SSHD_OPTS
ExecReload=/bin/kill -HUP $MAINPID
KillMode=process
Restart=on-failure

[Install]
WantedBy=multi-user.target
Alias=sshd.service

The unit files start with [Unit] section is where service information are defined. There are a brief description of service, dependencies (for example in that case systemd knows that couldn’t start this service before network and audit are running) and finally a condition to be checked before start (in this example if the path /etc/ssh/sshd_not_to_be_run exist, sshd will not be started).

The following section is [Service] where the information about what and how services are managed is defined. For example, notice that ExecStart is the command systemd will run to start this service. How to reload (ExecReload) and stop it (KillMode) is also defined.

EnvironmentFile is the default config file for the service but see that in this example the file path goes after “-” symbol. That symbol is not trivial, it says that if the file doesn’t exist ignore the errors and continue, you can read more about it in the official documentation5.

The KillMode defines how the services are stopped, in that case only the process named ssh will be stopped, nothing more, there’s other behaviours available, for example to kill all processes from the same cgroups group. You can read more about it in the man pages6.

In the last section named [Install] there’s the information about service location and alias which are a list of additional names for this service.

Ok, we’re going at a good pace!

We swears to serve the master of the services

Let’s go to see how we can manage these unit files we have in our system. To operate with systemd we shall use a tool named systemdctl. If we would like to modify system services remember to run every command with sudo or by any system user alter services status.

Starting, stopping and restarting services

To start a systemd service and run the instructions in the service’s unit file, we should use the start command:

systemctl start sshd.service

Notice that we can skip the “.service” because systemd will always look for “*.service” files for managing a service. So running this has the same behaviour:

systemctl start sshd

In the same way, to stop a service we will run:

systemctl stop ssh.service

or:

systemctl stop ssh

And even to restart or reload it:

systemctl restart ssh

and:

systemctl reload ssh
Enabling and Disabling services

Enable and disable are used to tell systemd to start services automatically at boot or not. So, to enable set automatically start on boot for a service we will simply run:

systemctl enable sshd

By the contrary if we want to disable auto starting we should run:

systemctl disable sshd
You shall not start!

Systemd defined not only one but three levels of stopped status:

  • Stopped: stop simply terminates the running instance of the specified service but the service could be started again manually or during boot process. We can perform such action running: systemctl stop ssh.service

  • Disabled: disabling a service will unhook it from starting boot process, it means that if we reboot the system this service will not be started. However we can still start it manually. To do that action we should run: systemctl disable ssh.service. Keep in mind that this process doesn’t stop the current running service.

  • Masked: finally, the last one goes one step further. It follows the same behaviour from disable state and ensures that the specified service couldn’t be started again by pointing de init hook of that service to /dev/null. This means that we can’t start it again. To do that we run: systemctl mask ssh.service. Notice that we can’t perform systemctl start ssh.service again.

The output has the following columns:

  • Unit: The unit name.

  • Load: The unit file was loaded by systemd and is kept in memory.

  • Active: A brief summary if the service was started successfully or not.

  • Sub: This indicates more detailed information about the unit which varies depending on unit type, state, and the actual method in which the unit runs.

  • Description: A description of what the service is/does.

Speak, journald, and enter

As we said, systemd has its own logging system called journald.

To read the full log we use:

journalctl

This command prints each boot, kernel and user land log files, but, of course, we can show one by one independently, for example:

To review the boot log:

journalctl -b

To review the kernel log:

journalctl -k

To get the log per one unit file:

journalctl -u ssh

For us there are two parameters that are very useful on our daily basis:

  • –pager-end (-e): This makes journalctl to immediately jump to the last lines. Very useful to see last events.

  • –follow (-f): Follow the log entries. This is the “A” letter from the journalctl alphabet, that follows the log stream in real time.
